# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

`pigeon` is a Zephyr RTOS module: the on-device client library for **PidgeIoT**. It runs on the physical asset/gateway and is the counterpart to the `dovecote` edge backend and `capsules` shared models in the main PidgeIoT monorepo at **`~/pidgeiot`**. This repo is a standalone Zephyr module (own git history, no path/workspace relationship to `~/pidgeiot`) — the two only need to agree on the wire protocol.

**When working on anything protocol-, auth-, or connector-shaped here, read `~/pidgeiot/CLAUDE.md` first and cross-check against these backend source files**, since this device library must stay wire-compatible with them:

- `~/pidgeiot/capsules/src/lib.rs` — the `Connector` enum (`Https(HttpsConfig)` / `Coap(CoapConfig)`) this library's `enum pigeon_connector_type` and `struct pigeon_config` must match the shape of. `HttpsConfig`/`CoapConfig` carry `endpoint` + `token` (+ DTLS PSK identity/secret for CoAP); on this side, `endpoint`/`token` are build-time `CONFIG_PIGEON_ENDPOINT`/`CONFIG_PIGEON_TOKEN` Kconfig strings rather than runtime struct fields — `PIGEON_CONNECTOR_TYPE` is already a build-time choice, so only one endpoint/token pair is ever live. PSK fields stay in `struct pigeon_coap_config` since they're optional per-device.
- `~/pidgeiot/dovecote/src/objects/pigeons.rs` (`get_shadow_device`) / `src/objects/helpers.rs` (`verify_device_token`) — devices authenticate with an opaque, **not-JWT** bearer credential, `Bearer`-prefixed: a 69-byte binary blob (`version | expires_at | Ed25519 signature`), base64url-encoded. It carries no pigeon-id claim of its own — the backend verifies it against whichever pigeon's own stored `device_public_key` the request's URL path resolves to, so this library never needs to embed or check its own identity beyond sending the right endpoint + token pair. (Rewritten 2026-07-15, replacing an earlier Ed25519-signed-JWT scheme (`require_device_auth`) that had two successive bugs — a `sub`-vs-`aud` claim mismatch, then a follow-on ACL check that 403'd every device request — both mooted by this rewrite rather than patched.) Whatever this library sends as `auth_token` must satisfy `verify_device_token`.
- `~/pidgeiot/dovecote/src/objects/pigeons.rs` (`build_http_endpoint`/`build_coap_endpoint`, and the `/pigeon/shadow/get` (dashboard), `/pigeon/device/shadow` (device, no `X-User-Id`/ACL check — see above), `/pigeon/shadow/update` routes) — defines the actual endpoint URL shapes (`https://api.pidgeiot.com/device/pigeons/{id}`, `coaps://api.pidgeiot.com/device/pigeons/{id}`) and the shadow document (`target_config`/`current_config` + version counters) this library syncs against.
- The backend issues/rotates the device token via `token/refresh`, which mints a **brand-new keypair** and overwrites the pigeon's stored public key — this revokes every previously-issued token for that pigeon, not just the one being replaced. Out of scope for this library to mint or verify tokens, only to present whatever it's given.
- **CoAP transport: resolved 2026-07-15.** Both sides now speak CoAP-over-TLS/TCP (RFC 8323, `coaps+tcp://`) — the backend switched `capsules::CoapConfig`/`build_coap_endpoint` from `dtls_psk_identity`/`dtls_psk_secret`/`coaps://` (UDP/DTLS) to match this library's `tls_psk_identity`/`tls_psk_secret` naming, since this device stack has no on-device UDP support. No outstanding gap here.

## Current state

`src/pigeon_core.c`: `pigeon_init` implemented; `pigeon_set_shadow_param` queues a single-slot pending key/val delta, flushed on demand by `pigeon_shadow_flush()` via the active transport's `pigeon_transport_report_shadow()`.

`src/pigeon_https.c` implements all three device-facing operations, each device-authenticated with `Authorization: Bearer <CONFIG_PIGEON_TOKEN>` via Zephyr's `http_client_req()`:
- `pigeon_shadow_get()` — GET `<CONFIG_PIGEON_ENDPOINT>/shadow`, decoded with the declarative JSON parser (`zephyr/data/json.h`) into `target_version`/`current_version`/`target_config`/`current_config`/`updated_at`. Requires `CONFIG_PIGEON_HTTPS_SEC_TAG` (default 1) to match whatever sec_tag the app provisioned its CA cert under.
- `pigeon_shadow_flush()` / `pigeon_transport_report_shadow()` — POST `<endpoint>/telemetry`, body `{"key":"val"}`, matching dovecote's `report_telemetry_device` (latest-value-per-key store).
- `pigeon_shadow_report()` — POST `<endpoint>/shadow`, body `{"current_config":<raw JSON>,"current_version":<int>}`, matching `capsules::PigeonShadowReportRequest`/dovecote's `report_shadow_device`. Call after applying a `target_config` fetched via `pigeon_shadow_get()`.

`src/pigeon_coap.c` mirrors all three operations over CoAP-over-TLS/TCP (`coaps+tcp://`, RFC 8323), using a shared `pigeon_coap_exchange()` with a `leaf` path segment (`"shadow"` vs `"telemetry"`) instead of separate builders per operation. The device bearer token rides in a Uri-Query option (CoAP has no header mechanism to mirror `Authorization: Bearer`). Wire-compatible with the backend since 2026-07-15 (see above); known real-hardware gap remains — PSK registration always calls `tls_credential_add()` with no `CONFIG_MODEM_KEY_MGMT` branch, so on real nRF91 hardware (TLS offloaded to the modem) PSK credentials never reach the modem's actual credential store. Needs a `modem_key_mgmt_write()` path for PSK before real-hardware CoAP auth works — see `pigeon-examples`' `coap_tcp_init` sample notes.

`src/pigeon_log_backend.c` (added 2026-07-17, `CONFIG_PIGEON_LOG_UPLOAD`): a Zephyr dictionary-mode log backend (`CONFIG_LOG_DICTIONARY_SUPPORT`) — format strings never leave the firmware image, only compact binary-encoded log records do. Buffers records in a ring, batches and uploads them over the active transport's HTTPS/CoAP POST to `<endpoint>/logs` (dovecote's device log ingestion route, 16KiB/chunk cap). Host-side decode needs the per-build `log_dictionary.json` artifact (produced by the Zephyr build, not checked in) plus `log_parser.py`'s `colorama` dependency (undocumented upstream gap, now noted in `pigeon-examples`' README). Opt-in via Kconfig — off by default.

`src/pigeon_fota.c` (added 2026-07-18, `CONFIG_PIGEON_FOTA`): the device half of OTA firmware updates, integrating with MCUboot. `pigeon_transport_download_firmware()` does the chunked HTTP Range GET against dovecote's `GET /device/pigeons/:id/firmware/:version` (matching the shadow's embedded `firmware` target — version/size/sha256), streaming straight into the MCUboot secondary slot rather than buffering the whole image in RAM. `pigeon_fota_apply()` verifies the assembled image's sha256, schedules the MCUboot test-swap, reports shadow convergence back to the platform, then triggers a graceful reboot (LTE detach first, same `CFUN=0` discipline as the rest of this library — see `pigeon-examples`' "Modem reset safety"). `pigeon_fota_confirm_boot()` must be called after the app decides the new image is healthy (`boot_write_img_confirmed()`); until that happens MCUboot will revert to the previous slot on the next N failed boots — this is the deliberate safety net, not a bug, and hasn't yet been exercised end-to-end on real hardware (see `pigeon-examples`' https_init FOTA wiring notes). **Known real-hardware gotcha (2026-07-20):** the default signing key used for MCUboot image signing in these samples is a dev/test key, not a production-grade one — documented as a real prod-readiness gap, not fixed, since rotating it is a deliberate deploy-time decision, not something to silently default away.

`src/pigeon_ws.c` (added 2026-07-20, `CONFIG_PIGEON_WS`, task #33, hardware-verified on ESP32-C6 against staging): a persistent `wss://<CONFIG_PIGEON_ENDPOINT>/ws` push channel (dovecote task #32's device WS endpoint), additive on top of the HTTPS connector (`depends on PIGEON_CONNECTOR_HTTPS` in Kconfig) rather than a third connector choice — WS never does shadow GET/report-back, log upload, or FOTA, HTTPS still owns those. Public API (`include/pigeon.h`): `pigeon_ws_start(pigeon_ws_event_cb_t cb)` spawns a dedicated worker thread that connects and reconnects forever with backoff; `pigeon_ws_stop()` sends a real CLOSE frame and joins the thread (call before `sys_reboot()`); `pigeon_ws_connected()` is a point-in-time query. The callback delivers `PIGEON_WS_EVENT_CONNECTED` (app should re-`pigeon_shadow_get()` since the server sends no state snapshot on accept), `PIGEON_WS_EVENT_DISCONNECTED` (informational, reconnect is automatic), and `PIGEON_WS_EVENT_SHADOW_UPDATE` (a dashboard `PUT` landed — `shadow` param points at module-static storage valid only for the callback's duration).

Frame protocol: outbound is `{"type":"ping"}` keepalives (device-owned — the server never pings, see `CONFIG_PIGEON_WS_PING_INTERVAL_SEC`'s help text) plus opportunistic `{"type":"telemetry","metrics":{...}}` reports; inbound is `{"type":"pong"}` (resets missed-pong bookkeeping; two consecutive misses mark the connection half-open and force a reconnect) and `{"type":"shadow_update","shadow":{...}}` (server push on a dashboard write). `pigeon_core.c`'s `pigeon_shadow_flush()` tries `pigeon_ws_report_telemetry()` first when `CONFIG_PIGEON_WS` is on, falling back to the HTTPS/CoAP transport on `-ENOTCONN` — telemetry-over-WS is fire-and-forget (no ack), so shadow_report itself never moves off HTTPS, where an HTTP status confirms persistence. Reconnect is exponential backoff from `PIGEON_WS_BACKOFF_BASE_SEC` (1s), doubling, capped at `CONFIG_PIGEON_WS_RECONNECT_MAX_DELAY_SEC` (default 300s), ±25% jitter, reset to base once a connection survives its first app-level pong. One socket per pigeon is enforced server-side (dovecote's `objects/ws.rs`); a rival connection stealing the slot closes this device's socket with code 4009, which this library treats like any other drop and reconnects into.

**Hard-won gotcha, preserved here on purpose**: `pigeon_ws_teardown()` always calls `websocket_disconnect()` itself, unconditionally, rather than trusting Zephyr's own `websocket_recv_msg()` to have already unref'd its context on a server-initiated CLOSE. That trust used to be the code, and it was wrong — the vendored `zephyr/subsys/net/lib/websocket/websocket.c` checks `ctx->message_type == WEBSOCKET_FLAG_CLOSE` (exact equality, not a bitwise `&`) before running its own internal-disconnect/unref path, but RFC 6455 sec 5.5.1 forbids fragmenting control frames, so any real CLOSE frame from a compliant peer also carries `WEBSOCKET_FLAG_FINAL`, making `message_type` `CLOSE|FINAL` (0x09) — which that exact-equality check can never match (0x09 != 0x08). Confirmed empirically (a native_sim harness with temporary ref/unref tracing on a scratch copy of the gitignored vendored file showed exactly one `ref()` and zero `unref()` calls across a full connect→server-close→reconnect cycle). In practice this makes the library's own internal-disconnect path dead code for every real peer, and silently leaked the sole `CONFIG_WEBSOCKET_MAX_CONTEXTS` slot on every server-initiated close — reproduced as a permanent post-churn `"upgrade handshake failed: -17"` (EEXIST) wedging every subsequent reconnect. Fixed by having `pigeon_ws_teardown()` call `websocket_disconnect()` (which unrefs via the *outer* close path, unaffected by the buggy check) itself every time, regardless of how the socket went down. Confirmed fixed on real ESP32-C6 hardware via a live socket-steal test (device saw the close, reconnected in ~1s, reclaimed the socket).

The `pigeon_json_escape()` shared helper (`pigeon_core.c`, declared in `pigeon_internal.h`) was promoted out of `pigeon_https.c`/`pigeon_coap.c` (which each carried an identical private copy) specifically because `pigeon_ws.c` would otherwise have needed a third copy; it's now RFC 8259 control-char compliant (0x00-0x1F, `\n`/`\r`/`\t` via shorthand, `\u00XX` otherwise) rather than only escaping `"`/`\` — found because a raw control byte in a telemetry value used to produce invalid JSON that HTTPS/CoAP could shrug off as one failed report, but over `pigeon_ws.c`'s shared persistent socket got the whole connection closed by dovecote's strict `serde_json` parse (code 4003), taking `shadow_update` push delivery down with it.

See `samples/https_init` in the sibling `pigeon-examples` repo for the HTTPS reference consumer (`src/shadow.c` fetches the shadow, applies `target_config` — app-defined `log`/`telemetry_interval`/`reboot` fields, the pigeon library itself doesn't parse `target_config`, see below — and polls on a loop driven by the shadow's own `telemetry_interval`), and `samples/wifi_init` for the `CONFIG_PIGEON_WS` reference consumer.

## Feature-parity gaps (noted 2026-07, not yet addressed)

Comparing against Circuit Dojo's "Lion" IoT server (https://www.circuitdojo.com/posts/introducing-lion-iot-server) and, more importantly, against what **AWS IoT customers** — the actual target market — already expect from AWS IoT Core (Device Shadow, Jobs, Device Defender, fleet provisioning). Lion's specific protocol choices are one data point, not a spec to copy verbatim; weigh each gap against the AWS IoT mental model, not Lion's.

Confirmed via `~/pidgeiot/capsules/src/lib.rs` and `~/pidgeiot/dovecote/src/lib.rs`: **none of the following exist on the backend yet** (grepped for `telemetry`, `generation`/`command`, `log_stream`/`dictionary`, `firmware`/`ota`/`block2` — zero hits). These are backend gaps, not just device-firmware gaps — device code can't usefully be written against endpoints that don't exist.

- ~~**Telemetry ingestion**: no endpoint at all.~~ **Resolved 2026-07-15**: `POST <endpoint>/telemetry` (`report_telemetry_device`, dovecote) + `pigeon_set_shadow_param()`/`pigeon_shadow_flush()` (device). Latest-value-per-key store, not a time-series log — a history/range-query table is a deliberately deferred follow-up, as is a `fancier` display view.
- ~~**Command execution with ack**: no device-side ack path back to the platform.~~ **Resolved 2026-07-15**: `POST <endpoint>/shadow` (`report_shadow_device`, dovecote) + `pigeon_shadow_report()` (device), called from `https_init`'s `shadow_sync()` after applying `target_config`. Still no formal command queue beyond `target_version`/`current_version` as an implicit generation counter (the `"reboot": true` one-shot-command pattern in `shadow.c`) — AWS IoT's Jobs service remains the closest analog worth studying if a real queue is wanted later.
- ~~**Log streaming**: no endpoint, no dictionary/token-compressed log encoding wired up.~~ **Resolved 2026-07-17**: `pigeon_log_backend.c` (`CONFIG_PIGEON_LOG_UPLOAD`, opt-in) + dovecote's `POST <endpoint>/logs` — dictionary-mode encoding, hardware-verified e2e (15 chunks decoded, byte-matched against serial). DO-local ring buffer only, no durable/Postgres-backed log store (deliberate — this is a rolling debug buffer).
- ~~**OTA / firmware rollout**: ... no remote fetch-and-flash path ... no backend endpoint to serve images or track fleet rollout state.~~ **Mostly resolved 2026-07-18-20, task #23-24**: backend has R2-backed firmware storage + Range-chunked device download + shadow-embedded firmware targets; `pigeon_fota.c` (`CONFIG_PIGEON_FOTA`) does the device-side download/verify/schedule/reboot. Hardware e2e (real nRF9160) got through download, sha256 verify, MCUboot test-swap scheduling, shadow convergence report, and graceful reboot — the still-open pieces are: confirming a post-reboot device re-authenticates and converges cleanly (blocked as of 2026-07-20 on a stale-token OTA test artifact, see dovecote notes in `~/pidgeiot/CLAUDE.md`), and the deliberate unconfirmed-image MCUboot-revert test (never yet exercised). Dev-signing-key-in-use for MCUboot image signing is a known, documented, not-yet-fixed prod-readiness gap.
- **Auth model**: current transport is HTTPS + a per-pigeon Ed25519 keypair with an opaque binary bearer token (see `get_shadow_device`/`verify_device_token`, not JWT as of 2026-07-15). Lion uses PSK+DTLS for bandwidth reasons on sleepy NB-IoT devices; AWS IoT's primary device auth is X.509 mutual TLS (though custom authorizers are supported). Worth deciding deliberately rather than drifting — per-device-keyed bearer tokens over HTTPS is a reasonable AWS-IoT-adjacent choice already, not obviously wrong.
- **RBAC**: dovecote has basic user-level auth (`require_auth`) and per-pigeon ACL rows, but no granular role model has been checked against what AWS IoT's Device Defender/policy model expects.
- ~~**Persistent/real-time channel**: backend-only, this library has no WS transport or client yet.~~ **Resolved 2026-07-20, task #33**: `pigeon_ws.c` (`CONFIG_PIGEON_WS`) implements the device side of dovecote's WebSocket endpoint (`GET /device/pigeons/:id/ws`, added 2026-07-19) — persistent connect/reconnect-with-backoff, device-owned ping keepalive, telemetry-over-WS with HTTPS fallback, and server-pushed `shadow_update` handling. Hardware-verified on real ESP32-C6 against staging, including a live socket-steal recovery test (see "Current state" above for the context-leak bug that test caught and fixed). Non-fatal ESP32 `wifi_adapter` heap allocation warnings remain open as a separate follow-up (PidgeIoT task #15), not blocking.

## Build

This is a Zephyr **module**, not a standalone buildable app — `CMakeLists.txt` hard-fails if `ZEPHYR_BASE` isn't set (see its guard clause). It must be built as a dependency from within a Zephyr application/workspace (`west build`), not compiled directly in this repo.

- `zephyr/module.yml` wires the module into a Zephyr manifest (`west.yml`) via `build.cmake`/`build.kconfig`.
- `zephyr/Kconfig` exposes `CONFIG_PIGEON` (menuconfig gate), a mutually-exclusive choice between `CONFIG_PIGEON_CONNECTOR_COAP` and `CONFIG_PIGEON_CONNECTOR_HTTPS`, `CONFIG_PIGEON_ENDPOINT`/`CONFIG_PIGEON_TOKEN` (string, required when `PIGEON` is enabled), `CONFIG_PIGEON_LOG_LEVEL` (0-4), and (`depends on PIGEON_CONNECTOR_HTTPS`) `CONFIG_PIGEON_WS` plus its sub-options (`_RX_BUF_SIZE`, `_PING_INTERVAL_SEC`, `_RECONNECT_MAX_DELAY_SEC`, `_THREAD_STACK_SIZE`) — reuses the HTTPS connector's endpoint/token/sec_tag, no separate WS credentials.
- `.vscode/settings.json` expects clangd compile commands at `build/https_init/compile_commands.json` — a symlink at `build` points to `~/pigeon-examples/build`, the shared build dir of the `https_init` sample in the sibling `pigeon-examples` repo (`samples/https_init`), which pulls this module in via `ZEPHYR_EXTRA_MODULES`.

## Conventions

- 2-space indentation (no `.clang-format` yet — follow the existing style in `src/pigeon_core.c`).
- Public API lives in `include/pigeon.h` behind an `extern "C"` guard; keep new public symbols there, not in `src/`.
- Use Zephyr's own logging (`LOG_MODULE_REGISTER`/`LOG_INF`/`LOG_ERR`), gated by `CONFIG_PIGEON_LOG_LEVEL`, not `printf`.
